此管線每日 UTC 時間上午 6 點透過 GitHub Actions (`validate-and-release.yml`) 執行，也會在每次推送至 `main` 且涉及 `scripts/`、`config/` 或 `pyproject.toml` 的變更時觸發。此外，也可手動觸發，並提供強制發佈或跳過即時 API 測試的選項。

## 管線階段

```
 Download         Validate          Spectral Lint
 ┌──────┐        ┌──────────┐      ┌─────────────┐
 │ Fetch │───────▶│Schemathesis│────▶│  Discover   │
 │ specs │        │Constraints│     │  violations │
 └──────┘        └──────────┘      └─────────────┘
                                          │
                 Spectral Gate       Reconcile
                ┌─────────────┐    ┌──────────┐
                │  Enforce    │◀───│Apply fixes│
                │  thresholds │    │to specs   │
                └─────────────┘    └──────────┘
                       │
                    Release
                 ┌──────────┐
                 │  Package  │
                 │  & tag    │
                 └──────────┘
```

### 1. 下載

`scripts/download.py` 從 `docs.cloud.f5.com` 取得官方 F5 XC OpenAPI 規格 ZIP 檔案。它使用 HTTP ETag 標頭進行條件式 GET 請求——若上游套件未變更，則跳過下載。解壓後的 JSON 規格檔案會存放於 `specs/original/`。

### 2. 驗證

`scripts/validate.py` 針對即時 F5 XC API 執行兩種測試：

- **Schemathesis** —— 基於屬性的測試，依據規格產生隨機的有效/無效負載，並檢查 API 是否符合規格的約束條件。
- **約束驗證** —— 針對全部 10 個 OpenAPI 約束類別進行的定向測試（字串長度、模式匹配、數值範圍、必填欄位、列舉值、陣列範圍、物件結構、組合、依賴關係、資料型別）。

差異結果會寫入 `reports/validation_report.json`。

### 3. Spectral 檢查（發現）

`scripts/spectral_lint.py --mode discover` 使用 `spectral-pipeline.yaml` 對原始規格執行 Spectral CLI。此階段會找出 OAS3 品質問題——缺少 servers 區塊、缺少聯絡資訊、未標記標籤的操作、未使用的元件結構描述、重複的 operationId、無效的範例，以及描述中的 script 標籤。

違規項目會轉換為 `Discrepancy` 物件並寫入 `reports/spectral_report.json`。完整規則集請參閱 [Spectral 規則](/04-spectral-rules/)。

### 4. 調和

`scripts/reconcile.py` 讀取兩份報告檔案，並對每個規格套用修正：

- **約束修正** 根據觀察到的 API 行為調整結構描述約束（放寬、收緊、新增、移除）。請參閱[已套用的修正](/03-fixes-applied/)。
- **Spectral 修正** 為規格補充 servers、聯絡資訊、標籤、安全性中繼資料，並移除無用的元件。
- **安全性中繼資料** 在 `security_scheme` 設定存在時一律注入，即使 Spectral 並未標記該問題。

修正後的規格會寫入 `release/specs/`，並在儲存前以 `openapi-spec-validator` 進行驗證。若修正後的規格驗證失敗，則會退回使用原始規格。

### 5. Spectral 把關

`scripts/spectral_lint.py --mode gate` 針對 `release/specs/` 中已調和的規格重新執行 Spectral。把關機制會根據可設定的閾值（`validation.yaml` 中的 `spectral.gate.max_errors` 和 `spectral.gate.max_warnings`）檢查錯誤與警告數量。

:::note
第一階段（目前）：閾值設為 `null`（僅警告模式）。設定 `max_errors: 0` 即可啟用零錯誤把關機制。
:::

### 6. 發佈

`scripts/release.py` 將調和後的規格打包為帶版本號的 ZIP 封存檔。版本號採用 `YYYY.MM.DD-<patch>` 格式，衍生自上游規格中繼資料的日期。發佈內容包括：

- 各領域獨立的規格檔案（268 個 JSON 檔案）
- 合併所有領域的 `openapi.json`
- 列出所有已套用修正的 `CHANGELOG.md`
- `VALIDATION_REPORT.md` 摘要

CI 工作會建立 GitHub Release 並附上 ZIP 檔案，標記為 `v<version>`。內容雜湊機制可防止在無任何變更時產生重複發佈。

## 文件更新

一個平行的 CI 工作會使用 `scripts/generate_docs.py` 從驗證報告產生 `docs/01-validation-report.mdx`，若內容有變更則會開啟 PR。

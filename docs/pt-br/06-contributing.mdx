## Configuração

```bash
# Clone e instale as dependências de desenvolvimento
git clone https://github.com/f5xc-salesdemos/api-specs.git
cd api-specs
make dev-install

# Instale o Spectral CLI (necessário para as etapas de linting)
npm install -g @stoplight/spectral-cli
```

`make dev-install` cria um virtualenv em `.venv/` e instala o projeto com extras de desenvolvimento (pytest, ruff, mypy).

### Variáveis de Ambiente

A validação de API em tempo real requer:

```bash
export F5XC_API_URL="https://example-tenant.console.ves.volterra.io"
export F5XC_API_TOKEN="your-api-token"
```

Sem essas variáveis, `make validate` utiliza o modo dry-run como fallback.

## Executando o Pipeline

### Pipeline Completo

```bash
make all
# Executa: download → validate → spectral-lint → reconcile → spectral-gate → release
```

### Etapas Individuais

```bash
make download        # Busca as specs upstream
make validate        # Executa testes de API em tempo real (requer token de API)
make validate-dry    # Execução simulada (sem chamadas de API)
make spectral-lint   # Modo discover do Spectral nas specs originais
make reconcile       # Aplica correções de ambos os relatórios
make spectral-gate   # Quality gate do Spectral nas specs reconciliadas
make release         # Gera o pacote de release
```

### Documentação

```bash
make docs-generate   # Regenera docs/01-validation-report.mdx a partir dos relatórios
make docs            # Gera o site de documentação completo
make docs-serve      # Servidor de desenvolvimento local com hot reload
```

## Executando os Testes

```bash
make test           # pytest com cobertura
make lint           # ruff check + verificação de formatação
make typecheck      # verificação de tipos com mypy
make ci-test        # Todos os três (usado no CI)
```

Os hooks de pre-commit são executados automaticamente no commit, se instalados:

```bash
make pre-commit-install   # Instala os git hooks
make pre-commit           # Executa todos os hooks manualmente
```

## Adicionando um Novo Método de Correção

Para adicionar uma nova correção automática do Spectral ao reconciliador:

1. **Adicione a regra** ao `.spectral.yaml` com a severidade desejada.

2. **Habilite a correção automática** em `config/validation.yaml` em `spectral.auto_fix`:
   ```yaml
   spectral:
     auto_fix:
       your-new-rule: true
   ```

3. **Adicione o método de correção** ao `SpecReconciler` em `scripts/reconcile.py`:
   ```python
   def _fix_your_rule(self, spec: dict, discrepancy: Discrepancy) -> dict | None:
       """Fix description."""
       # Modify spec in place
       # Return spec if changed, None if no change needed
       return spec
   ```

4. **Registre-o** no dispatcher `_apply_spectral_fix`:
   ```python
   spectral_fixers = {
       # ... existing fixers ...
       "spectral:your-new-rule": self._fix_your_rule,
   }
   ```

5. **Adicione testes** cobrindo o novo método de correção.

## Adicionando uma Nova Regra do Spectral

Para adicionar uma regra customizada do Spectral com uma função JavaScript:

1. Crie a função em `spectral/functions/your-rule.js`. Exporte uma função padrão que receba `(targetVal, options, context)` e retorne um array de objetos `{message, path}`.

2. Registre-a em `spectral-pipeline.yaml`:
   ```yaml
   functions:
     - f5-path-params
     - your-rule

   rules:
     your-rule:
       description: "What it checks"
       message: "{{error}}"
       severity: error
       given: "$.paths[*]"
       then:
         function: your-rule
   ```

:::caution
Funções customizadas funcionam apenas em `spectral-pipeline.yaml`. Não adicione `functionsDir` ou `functions` ao `.spectral.yaml` -- essa configuração deve permanecer compatível com o Super-Linter e outras ferramentas externas.
:::

## Pipeline de CI

O workflow do GitHub Actions (`validate-and-release.yml`) é executado em:

- **Agendamento**: Diariamente às 6h UTC
- **Push para main**: Quando há alterações em `scripts/`, `config/` ou `pyproject.toml`
- **Execução manual**: Com opções para release forçada ou dry-run

O workflow possui quatro jobs:

1. **validate** -- Baixa as specs, executa validação, lint do Spectral, reconciliação, quality gate do Spectral
2. **check-release-needed** -- Compara hashes de conteúdo para decidir se uma release é necessária
3. **release** -- Empacota e cria uma Release no GitHub (apenas se o conteúdo mudou)
4. **update-docs** -- Regenera `01-validation-report.mdx` e abre um PR se houver alterações

Um job **notify** cria uma issue no GitHub se algum job falhar.

## Fluxo de Branches e PRs

Todas as alterações seguem: **Issue -> Branch -> PR -> CI aprovado -> Merge**.

- Nomes de branches: `feature/<issue>-desc`, `fix/<issue>-desc`, `docs/<issue>-desc`
- PRs devem referenciar uma issue (`Closes #N` na descrição)
- Verificações de CI obrigatórias: `Check linked issues` e `Lint Code Base`
- Squash merge é preferido; branches são excluídas automaticamente após o merge

Consulte `CONTRIBUTING.md` para as regras completas do fluxo de trabalho.

协调器（`scripts/reconcile.py`）对上游规范应用四类修复。约束修复会更改 API 契约以匹配观察到的行为。所有其他修复仅涉及元数据，不会更改 API 契约。

## 约束修复

这些修复调整 OpenAPI 模式约束，使规范与实际 API 的行为保持一致。每项修复由[验证](/02-pipeline/#2-validate)过程中产生的 `Discrepancy` 对象驱动。

| 策略 | 触发条件 | 执行操作 | 契约变更 |
|----------|---------|--------------|-----------------|
| **放宽** | `SPEC_STRICTER` | 规范拒绝了 API 接受的值。降低 `minLength`/`minimum`，提高 `maxLength`/`maximum`，添加缺失的枚举值。 | 是 -- 扩大接受值范围 |
| **收紧** | `SPEC_LOOSER` | 规范接受了 API 拒绝的值。提高 `minLength`/`minimum`，降低 `maxLength`/`maximum`，将枚举限制为观察到的值。 | 是 -- 缩小接受值范围 |
| **添加** | `MISSING_CONSTRAINT` | API 强制执行了规范中未记录的约束。将该约束添加到模式中。 | 是 -- 添加新约束 |
| **移除** | `EXTRA_CONSTRAINT` | 规范声明了 API 忽略的约束。从模式中移除该约束。 | 是 -- 移除约束 |

:::caution
约束修复会更改 API 契约。仅在有来自 Schemathesis 或针对实际 API 的约束验证测试的证据支持时才会应用。
:::

### 支持的约束类型

测试和修复的 10 个 OpenAPI 约束类别：

- **字符串长度** -- `minLength`、`maxLength`
- **模式** -- `pattern`（正则表达式）
- **数值范围** -- `minimum`、`maximum`、`exclusiveMinimum`、`exclusiveMaximum`
- **必填字段** -- `required`
- **枚举值** -- `enum`
- **数组范围** -- `minItems`、`maxItems`、`uniqueItems`
- **对象结构** -- `additionalProperties`、`properties`、`propertyNames`
- **组合** -- `oneOf`、`anyOf`、`allOf`
- **依赖关系** -- `dependentRequired`、`dependentSchemas`
- **数据类型** -- `type`、`format`

## Spectral 增强

这些修复解决由 [Spectral 检查](/04-spectral-rules/)发现的 OAS3 质量问题。它们添加或修正元数据，不会更改 API 行为。

### 服务器块注入

**规则**：`oas3-api-servers` | **契约变更**：无

为缺少 `servers` 块的规范添加 F5 XC 租户 URL 模板：

```json
"servers": [{
  "url": "https://{tenant}.console.ves.volterra.io",
  "description": "F5 Distributed Cloud API",
  "variables": {
    "tenant": {
      "default": "example-tenant",
      "description": "Your F5 XC tenant name"
    }
  }
}]
```

服务器 URL 和变量值来自 [validation.yaml](/05-configuration/) 中的 `spectral.servers`。

### 联系信息注入

**规则**：`info-contact` | **契约变更**：无

为缺少联系信息的规范在 `info.contact` 中添加联系信息：

```json
"contact": {
  "name": "F5 Distributed Cloud",
  "url": "https://docs.cloud.f5.com",
  "email": "support@f5.com"
}
```

值来自 [validation.yaml](/05-configuration/) 中的 `spectral.contact`。

### 操作标签派生

**规则**：`operation-tags` | **契约变更**：无

为未标记的操作从 URL 路径前缀派生标签。对于类似 `/api/config/namespaces/{namespace}/healthchecks` 的路径，标签为 `config`（第二个非参数路径段）。该标签会同时添加到操作的 `tags` 数组和规范级别的 `tags` 列表中。

### 未使用组件移除

**规则**：`oas3-unused-component` | **契约变更**：无

从 `components.schemas` 中移除规范中任何位置都未引用的组件模式。减少规范大小且不影响任何操作。

### OperationId 去重

**规则**：`operation-operationId-unique` | **契约变更**：无

当多个操作共享相同的 `operationId` 时，将 HTTP 方法作为后缀追加（例如 `ListItems_get`）。这使每个 operationId 对于代码生成器来说都是唯一的。

### 无效示例/默认值移除

**规则**：`oas3-valid-schema-example` | **契约变更**：无

当 `example` 或 `default` 值不符合模式自身的约束（类型错误、超出范围等）时，将其从模式中移除。防止代码生成器生成无效的示例载荷。

### Script 标签清除

**规则**：`no-script-tags-in-markdown` | **契约变更**：无

使用正则替换从 `description` 字段中清除 `<script>` 标签。某些上游规范包含注入的 script 标签，会导致文档渲染器无法正常工作。

## 安全元数据

**契约变更**：无（仅文档）

每个规范如果尚未包含 `apiKeyAuth` 安全方案，都会被注入一个。无论 Spectral 是否标记了该问题，此项都会始终注入。

```json
"components": {
  "securitySchemes": {
    "apiKeyAuth": {
      "type": "apiKey",
      "in": "header",
      "name": "Authorization",
      "description": "F5 XC API Token (format: APIToken <token>)"
    }
  }
},
"security": [{ "apiKeyAuth": [] }]
```

方案定义来自 [validation.yaml](/05-configuration/) 中的 `spectral.security_scheme`。这告诉 API 使用者如何进行身份验证，但不会更改运行时行为。

## JSON 格式化

**契约变更**：无

所有输出的 JSON 规范都会通过与 Biome 兼容的数组压缩器（`scripts/utils/spec_loader.py` 中的 `_compact_short_arrays`）处理。可以在 120 个字符内容纳的短数组会被折叠为单行：

```json
// Before
"enum": [
  "ACTIVE",
  "INACTIVE"
]

// After
"enum": ["ACTIVE", "INACTIVE"]
```

超过行长度阈值的数组保持多行格式。

## 修复优先级

当多种修复策略可能适用时，协调器使用 `reconciliation.priority` 中配置的优先级顺序：

1. **existing** -- 如果现有规范约束有效，则使用现有约束
2. **discovery** -- 使用发现的实际 API 行为
3. **inferred** -- 从类似端点的模式中推断

所有修复应用完成后，每个修复后的规范都会通过 `openapi-spec-validator` 进行验证。如果验证失败，将使用原始规范作为回退，并将该修复记录为错误。

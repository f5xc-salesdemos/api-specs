## การตั้งค่า

```bash
# Clone and install dev dependencies
git clone https://github.com/f5xc-salesdemos/api-specs.git
cd api-specs
make dev-install

# Install Spectral CLI (required for linting stages)
npm install -g @stoplight/spectral-cli
```

`make dev-install` สร้าง virtualenv ใน `.venv/` และติดตั้งโปรเจกต์พร้อม dev extras (pytest, ruff, mypy)

### ตัวแปรสภาพแวดล้อม

การตรวจสอบ API แบบ Live ต้องการ:

```bash
export F5XC_API_URL="https://example-tenant.console.ves.volterra.io"
export F5XC_API_TOKEN="your-api-token"
```

หากไม่มีค่าเหล่านี้ `make validate` จะย้อนกลับไปใช้โหมด dry-run

## การเรียกใช้ Pipeline

### Pipeline แบบเต็ม

```bash
make all
# Runs: download → validate → spectral-lint → reconcile → spectral-gate → release
```

### ขั้นตอนแต่ละรายการ

```bash
make download        # Fetch upstream specs
make validate        # Run live API tests (requires API token)
make validate-dry    # Dry run (no API calls)
make spectral-lint   # Spectral discover mode on original specs
make reconcile       # Apply fixes from both reports
make spectral-gate   # Spectral quality gate on reconciled specs
make release         # Build release package
```

### เอกสารประกอบ

```bash
make docs-generate   # Regenerate docs/01-validation-report.mdx from reports
make docs            # Build full documentation site
make docs-serve      # Local dev server with hot reload
```

## การเรียกใช้การทดสอบ

```bash
make test           # pytest with coverage
make lint           # ruff check + format check
make typecheck      # mypy type checking
make ci-test        # All three (used in CI)
```

Pre-commit hooks จะทำงานโดยอัตโนมัติเมื่อ commit หากติดตั้งแล้ว:

```bash
make pre-commit-install   # Install git hooks
make pre-commit           # Run all hooks manually
```

## การเพิ่ม Fix Method ใหม่

ในการเพิ่ม Spectral auto-fix ใหม่ให้กับ reconciler:

1. **เพิ่มกฎ** ใน `.spectral.yaml` พร้อมระดับความรุนแรงที่ต้องการ

2. **เปิดใช้งาน auto-fix** ใน `config/validation.yaml` ภายใต้ `spectral.auto_fix`:
   ```yaml
   spectral:
     auto_fix:
       your-new-rule: true
   ```

3. **เพิ่ม fixer method** ใน `SpecReconciler` ในไฟล์ `scripts/reconcile.py`:
   ```python
   def _fix_your_rule(self, spec: dict, discrepancy: Discrepancy) -> dict | None:
       """Fix description."""
       # Modify spec in place
       # Return spec if changed, None if no change needed
       return spec
   ```

4. **ลงทะเบียน** ใน `_apply_spectral_fix` dispatcher:
   ```python
   spectral_fixers = {
       # ... existing fixers ...
       "spectral:your-new-rule": self._fix_your_rule,
   }
   ```

5. **เพิ่มการทดสอบ** ที่ครอบคลุม fix method ใหม่

## การเพิ่ม Spectral Rule ใหม่

ในการเพิ่ม Spectral rule แบบกำหนดเองพร้อมฟังก์ชัน JavaScript:

1. สร้างฟังก์ชันใน `spectral/functions/your-rule.js` ส่งออกฟังก์ชัน default ที่รับ `(targetVal, options, context)` และส่งคืนอาร์เรย์ของอ็อบเจกต์ `{message, path}`

2. ลงทะเบียนใน `spectral-pipeline.yaml`:
   ```yaml
   functions:
     - f5-path-params
     - your-rule

   rules:
     your-rule:
       description: "What it checks"
       message: "{{error}}"
       severity: error
       given: "$.paths[*]"
       then:
         function: your-rule
   ```

:::caution
ฟังก์ชันแบบกำหนดเองใช้งานได้เฉพาะใน `spectral-pipeline.yaml` เท่านั้น อย่าเพิ่ม `functionsDir` หรือ `functions` ใน `.spectral.yaml` -- การกำหนดค่านั้นต้องยังคงเข้ากันได้กับ Super-Linter และเครื่องมือภายนอกอื่นๆ
:::

## CI Pipeline

GitHub Actions workflow (`validate-and-release.yml`) ทำงานเมื่อ:

- **กำหนดเวลา**: ทุกวันเวลา 6 AM UTC
- **Push ไปยัง main**: เมื่อ `scripts/`, `config/`, หรือ `pyproject.toml` มีการเปลี่ยนแปลง
- **การเรียกใช้ด้วยตนเอง**: พร้อมตัวเลือกสำหรับการ release แบบบังคับหรือ dry-run

Workflow มีสี่ jobs:

1. **validate** -- ดาวน์โหลด specs, เรียกใช้การตรวจสอบ, Spectral lint, reconcile, Spectral gate
2. **check-release-needed** -- เปรียบเทียบ content hashes เพื่อตัดสินใจว่าจำเป็นต้อง release หรือไม่
3. **release** -- แพ็กเกจและสร้าง GitHub Release (เฉพาะเมื่อเนื้อหามีการเปลี่ยนแปลง)
4. **update-docs** -- สร้าง `01-validation-report.mdx` ใหม่และเปิด PR หากมีการเปลี่ยนแปลง

Job **notify** จะสร้าง GitHub issue หาก job ใดล้มเหลว

## ขั้นตอนการทำงานของ Branch และ PR

การเปลี่ยนแปลงทั้งหมดดำเนินตาม: **Issue -> Branch -> PR -> CI ผ่าน -> Merge**

- ชื่อ Branch: `feature/<issue>-desc`, `fix/<issue>-desc`, `docs/<issue>-desc`
- PR ต้องเชื่อมโยงกับ issue (`Closes #N` ในคำอธิบาย)
- CI checks ที่จำเป็น: `Check linked issues` และ `Lint Code Base`
- แนะนำ squash merge; branches จะถูกลบโดยอัตโนมัติหลัง merge

ดู `CONTRIBUTING.md` สำหรับกฎขั้นตอนการทำงานฉบับสมบูรณ์

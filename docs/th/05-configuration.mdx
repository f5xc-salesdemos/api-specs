พฤติกรรมทั้งหมดของไปป์ไลน์ถูกควบคุมโดยไฟล์ YAML สองไฟล์ใน `config/`

## `config/validation.yaml`

### การตั้งค่า API

```yaml
api:
  base_url: "https://example-tenant.console.ves.volterra.io"
  tenant: "example-tenant"
  namespace: "example-namespace"
  timeout: 30
  retries: 3
  retry_delay: 2
```

| ฟิลด์ | คำอธิบาย |
|-------|-------------|
| `base_url` | URL ฐานของ API คอนโซล F5 XC สามารถแทนที่ด้วยตัวแปรสภาพแวดล้อม `F5XC_API_URL` |
| `tenant` | ชื่อ Tenant ที่ใช้ในเส้นทาง API |
| `namespace` | Namespace เริ่มต้นสำหรับการดำเนินการ CRUD |
| `timeout` | ระยะเวลาหมดเวลาของคำขอ HTTP เป็นวินาที |
| `retries` | จำนวนครั้งที่ลองใหม่เมื่อเกิดข้อผิดพลาด |
| `retry_delay` | จำนวนวินาทีระหว่างการลองใหม่แต่ละครั้ง |

:::caution
ค่า `base_url`, `tenant` และ `namespace` ใน `validation.yaml` เป็นค่าตัวอย่างทั่วไป คุณ**ต้อง**กำหนดค่าเหล่านี้สำหรับสภาพแวดล้อมของคุณผ่านตัวแปรสภาพแวดล้อมก่อนเรียกใช้ไปป์ไลน์:

```bash
export F5XC_API_URL="https://example-tenant.console.ves.volterra.io"
export F5XC_TENANT="example-tenant"
export F5XC_NAMESPACE="example-namespace"
export F5XC_API_TOKEN="your-api-token"
```

ตัวแปรสภาพแวดล้อมจะมีความสำคัญเหนือกว่าค่าเริ่มต้นใน YAML ในขณะรันไทม์
:::

### การตั้งค่าการดาวน์โหลด

```yaml
download:
  url: "https://docs.cloud.f5.com/docs-v2/downloads/f5-distributed-cloud-open-api.zip"
  output_dir: "specs/original"
  etag_cache: ".etag_cache"
```

| ฟิลด์ | คำอธิบาย |
|-------|-------------|
| `url` | URL ของชุด OpenAPI spec อย่างเป็นทางการของ F5 XC (ZIP) |
| `output_dir` | ตำแหน่งที่จัดเก็บ spec ที่แตกไฟล์แล้ว |
| `etag_cache` | ไฟล์ที่จัดเก็บ HTTP ETag สำหรับการดาวน์โหลดแบบมีเงื่อนไข |

### หมวดหมู่การตรวจสอบ

สิบหมวดหมู่ที่สอดคล้องกับประเภทข้อจำกัดของ OpenAPI schema แต่ละหมวดหมู่สามารถเปิด/ปิดใช้งานได้อย่างอิสระ

```yaml
validation_categories:
  string_length:
    enabled: true
    keywords: ["minLength", "maxLength"]
    description: "Validate string length boundaries"
```

| หมวดหมู่ | คีย์เวิร์ด |
|----------|----------|
| `string_length` | `minLength`, `maxLength` |
| `pattern` | `pattern` |
| `numeric_bounds` | `minimum`, `maximum`, `exclusiveMinimum`, `exclusiveMaximum` |
| `required_fields` | `required` |
| `enum_values` | `enum` |
| `array_bounds` | `minItems`, `maxItems`, `uniqueItems` |
| `object_structure` | `additionalProperties`, `properties`, `propertyNames` |
| `composition` | `oneOf`, `anyOf`, `allOf` |
| `dependencies` | `dependentRequired`, `dependentSchemas` |
| `data_types` | `type`, `format` |

### การตั้งค่า Schemathesis

```yaml
schemathesis:
  enabled: true
  max_examples: 100
  hypothesis_phases:
    - generate
    - target
  stateful_testing: true
  base_url_override: null
  auth_header: "Authorization"
  auth_prefix: "APIToken"
```

| ฟิลด์ | คำอธิบาย |
|-------|-------------|
| `enabled` | เปิด/ปิดการทดสอบ Schemathesis |
| `max_examples` | จำนวนสูงสุดของกรณีทดสอบที่สร้างขึ้นต่อการดำเนินการหนึ่งรายการ |
| `hypothesis_phases` | เฟสของ Hypothesis ที่จะรัน (`generate`, `target`) |
| `stateful_testing` | เปิดใช้งานการทดสอบแบบ stateful ที่ใช้ลิงก์ข้ามการดำเนินการ |
| `base_url_override` | แทนที่ URL ฐานของ API สำหรับการทดสอบ `null` จะใช้ `api.base_url` |
| `auth_header` | ชื่อ HTTP header สำหรับการยืนยันตัวตน |
| `auth_prefix` | รูปแบบคำนำหน้าของโทเค็น (คำขอจะถูกส่งในรูปแบบ `APIToken <token>`) |

### การตั้งค่าการกระทบยอด

```yaml
reconciliation:
  priority:
    - existing
    - discovery
    - inferred
  fix_strategies:
    tighter_spec: "relax"
    looser_spec: "tighten"
    missing_constraint: "add"
    extra_constraint: "remove"
```

| ฟิลด์ | คำอธิบาย |
|-------|-------------|
| `priority` | ลำดับความสำคัญเมื่อแหล่งข้อมูลหลายแหล่งไม่ตรงกัน |
| `fix_strategies` | แมปประเภทความคลาดเคลื่อนแต่ละประเภทไปยังการดำเนินการแก้ไข ดูที่ [การแก้ไขที่ใช้](/03-fixes-applied/) |

### การตั้งค่า Spectral

```yaml
spectral:
  enabled: true
  ruleset: "spectral-pipeline.yaml"
  auto_fix:
    oas3-api-servers: true
    info-contact: true
    operation-tags: true
    oas3-unused-component: true
    operation-operationId-unique: true
    oas3-valid-schema-example: true
    no-script-tags-in-markdown: true
  gate:
    max_errors: null
    max_warnings: null
  contact:
    name: "F5 Distributed Cloud"
    url: "https://docs.cloud.f5.com"
    email: "support@f5.com"
  servers:
    - url: "https://{tenant}.console.ves.volterra.io"
      description: "F5 Distributed Cloud API"
      variables:
        tenant:
          default: "example-tenant"
          description: "Your F5 XC tenant name"
  security_scheme:
    type: "apiKey"
    in: "header"
    name: "Authorization"
    description: "F5 XC API Token (format: APIToken <token>)"
```

| ฟิลด์ | คำอธิบาย |
|-------|-------------|
| `enabled` | เปิด/ปิดการตรวจสอบ Spectral linting ทั้งหมด |
| `ruleset` | ไฟล์การกำหนดค่า Spectral ที่จะใช้ (ไปป์ไลน์ใช้ `spectral-pipeline.yaml`) |
| `auto_fix` | แมปชื่อกฎไปยังค่าบูลีน `true` หมายความว่าตัวกระทบยอดจะแก้ไขการละเมิด |
| `gate.max_errors` | จำนวนข้อผิดพลาดสูงสุดที่อนุญาตในเกทหลังการกระทบยอด `null` จะปิดใช้งานการตรวจสอบ |
| `gate.max_warnings` | จำนวนคำเตือนสูงสุดที่อนุญาต `null` จะปิดใช้งานการตรวจสอบ |
| `contact` | ข้อมูลติดต่อที่จะถูกแทรกเข้าไปใน `info.contact` โดยตัวแก้ไข `info-contact` |
| `servers` | อาร์เรย์ servers ที่จะถูกแทรกโดยตัวแก้ไข `oas3-api-servers` |
| `security_scheme` | นิยาม security scheme ที่จะถูกแทรกเข้าไปในทุก spec |

### การตั้งค่าการเผยแพร่

```yaml
release:
  output_dir: "release"
  include_changelog: true
  include_validation_report: true
  version_from: "git"
```

| ฟิลด์ | คำอธิบาย |
|-------|-------------|
| `output_dir` | ไดเรกทอรีสำหรับอาร์ทิแฟกต์การเผยแพร่ |
| `include_changelog` | รวม `CHANGELOG.md` ในแพ็กเกจการเผยแพร่ |
| `include_validation_report` | รวมรายงานการตรวจสอบในแพ็กเกจการเผยแพร่ |
| `version_from` | แหล่งที่มาของเวอร์ชัน `git` จะดึงเวอร์ชันจากวันที่เมตาดาต้าของ spec + หมายเลขแพตช์ |

## `config/endpoints.yaml`

กำหนดจุดปลายทางพื้นฐานที่ใช้สำหรับการตรวจสอบ API แบบสด แต่ละรายการจะแมปชื่อทรัพยากรเชิงตรรกะไปยังไฟล์ spec, กลุ่ม API และเส้นทาง CRUD

### โครงสร้างจุดปลายทาง

```yaml
endpoints:
  healthcheck:
    resource: healthchecks
    domain_file: docs-cloud-f5-com.0124.public.ves.io.schema.healthcheck.ves-swagger.json
    api_group: config
    crud_operations:
      create: POST /api/config/namespaces/{namespace}/healthchecks
      read: GET /api/config/namespaces/{namespace}/healthchecks/{name}
      list: GET /api/config/namespaces/{namespace}/healthchecks
      update: PUT /api/config/namespaces/{namespace}/healthchecks/{name}
      delete: DELETE /api/config/namespaces/{namespace}/healthchecks/{name}
    test_priority: high
    description: "Health check configurations for origin monitoring"
```

| ฟิลด์ | คำอธิบาย |
|-------|-------------|
| `resource` | ชื่อทรัพยากร API (ใช้ในเส้นทาง URL) |
| `domain_file` | ชื่อไฟล์ของ OpenAPI spec ใน `specs/original/` |
| `api_group` | คำนำหน้ากลุ่ม API (`config`, `web` ฯลฯ) |
| `crud_operations` | เมธอด HTTP และเส้นทางสำหรับการดำเนินการ CRUD แต่ละรายการ |
| `test_priority` | `high`, `medium` หรือ `low` -- ควบคุมลำดับการรันการทดสอบ |
| `description` | คำอธิบายทรัพยากรที่มนุษย์อ่านได้ |

### จุดปลายทางพื้นฐานปัจจุบัน

10 จุดปลายทางข้ามสามโดเมน:

| จุดปลายทาง | โดเมน | ความสำคัญ |
|----------|--------|----------|
| `healthcheck` | Virtual | high |
| `origin_pool` | Virtual | high |
| `app_firewall` | Virtual | high |
| `service_policy` | Virtual | medium |
| `api_definition` | API Security | high |
| `api_discovery` | API Security | medium |
| `api_groups` | API Security | medium |
| `code_base_integration` | API Security | low |
| `data_type` | Data Privacy | medium |
| `sensitive_data_policy` | Data Privacy | medium |

### การเพิ่มจุดปลายทางใหม่

เพื่อเพิ่มจุดปลายทางใหม่สำหรับการตรวจสอบ:

1. ค้นหาชื่อไฟล์ spec ใน `specs/original/` (รูปแบบ: `docs-cloud-f5-com.NNNN.*.ves-swagger.json`)
2. เพิ่มรายการภายใต้ `endpoints:` ตามโครงสร้างด้านบน
3. เพิ่มการแมปไฟล์โดเมนภายใต้ `domain_files:` พร้อมหมายเลขลำดับความสำคัญ
4. เพิ่มชื่อจุดปลายทางไปยัง `test_order:` ในตำแหน่งที่ต้องการ

:::tip
เฟรมเวิร์กจะตรวจสอบ spec ทั้งหมด 268 ไฟล์ระหว่างการกระทบยอด โดยไม่คำนึงว่ามีอะไรระบุไว้ใน `endpoints.yaml` การกำหนดค่าจุดปลายทางจะควบคุมว่าทรัพยากรใดจะได้รับการทดสอบ API แบบสดด้วย Schemathesis
:::

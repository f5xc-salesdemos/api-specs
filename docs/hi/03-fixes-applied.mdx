रीकन्सिलर (`scripts/reconcile.py`) अपस्ट्रीम स्पेक्स पर चार श्रेणियों के सुधार लागू करता है। कन्स्ट्रेंट सुधार API कॉन्ट्रैक्ट्स को देखे गए व्यवहार से मिलान करने के लिए बदलते हैं। अन्य सभी सुधार केवल मेटाडेटा से संबंधित हैं और API कॉन्ट्रैक्ट्स में कोई परिवर्तन नहीं करते।

## कन्स्ट्रेंट सुधार

ये सुधार OpenAPI स्कीमा कन्स्ट्रेंट्स को समायोजित करते हैं ताकि स्पेक लाइव API द्वारा वास्तव में लागू किए जाने वाले नियमों से मेल खाए। प्रत्येक सुधार [सत्यापन](/02-pipeline/#2-validate) के दौरान उत्पन्न `Discrepancy` ऑब्जेक्ट द्वारा संचालित होता है।

| रणनीति | ट्रिगर | यह क्या करता है | कॉन्ट्रैक्ट परिवर्तन |
|----------|---------|--------------|-----------------|
| **शिथिल करें (Relax)** | `SPEC_STRICTER` | स्पेक उन मानों को अस्वीकार करता है जिन्हें API स्वीकार करता है। `minLength`/`minimum` घटाता है, `maxLength`/`maximum` बढ़ाता है, छूटे हुए enum मान जोड़ता है। | हाँ -- स्वीकृत मानों को विस्तृत करता है |
| **कठोर करें (Tighten)** | `SPEC_LOOSER` | स्पेक उन मानों को स्वीकार करता है जिन्हें API अस्वीकार करता है। `minLength`/`minimum` बढ़ाता है, `maxLength`/`maximum` घटाता है, enums को देखे गए मानों तक सीमित करता है। | हाँ -- स्वीकृत मानों को संकुचित करता है |
| **जोड़ें (Add)** | `MISSING_CONSTRAINT` | API एक ऐसा कन्स्ट्रेंट लागू करता है जो स्पेक में प्रलेखित नहीं है। स्कीमा में कन्स्ट्रेंट जोड़ता है। | हाँ -- नया कन्स्ट्रेंट जोड़ता है |
| **हटाएँ (Remove)** | `EXTRA_CONSTRAINT` | स्पेक एक ऐसा कन्स्ट्रेंट घोषित करता है जिसे API अनदेखा करता है। स्कीमा से कन्स्ट्रेंट हटाता है। | हाँ -- कन्स्ट्रेंट हटाता है |

:::caution
कन्स्ट्रेंट सुधार API कॉन्ट्रैक्ट को बदलते हैं। ये केवल तभी लागू किए जाते हैं जब लाइव API के विरुद्ध Schemathesis या कन्स्ट्रेंट सत्यापन परीक्षण से साक्ष्य प्राप्त होता है।
:::

### समर्थित कन्स्ट्रेंट प्रकार

परीक्षित और सुधारी जाने वाली 10 OpenAPI कन्स्ट्रेंट श्रेणियाँ:

- **स्ट्रिंग लंबाई** -- `minLength`, `maxLength`
- **पैटर्न** -- `pattern` (regex)
- **संख्यात्मक सीमाएँ** -- `minimum`, `maximum`, `exclusiveMinimum`, `exclusiveMaximum`
- **आवश्यक फ़ील्ड** -- `required`
- **Enum मान** -- `enum`
- **ऐरे सीमाएँ** -- `minItems`, `maxItems`, `uniqueItems`
- **ऑब्जेक्ट संरचना** -- `additionalProperties`, `properties`, `propertyNames`
- **कम्पोज़िशन** -- `oneOf`, `anyOf`, `allOf`
- **डिपेंडेंसीज़** -- `dependentRequired`, `dependentSchemas`
- **डेटा प्रकार** -- `type`, `format`

## Spectral संवर्धन

ये सुधार [Spectral लिंटिंग](/04-spectral-rules/) द्वारा पाई गई OAS3 गुणवत्ता समस्याओं को संबोधित करते हैं। ये API व्यवहार बदले बिना मेटाडेटा जोड़ते या सही करते हैं।

### सर्वर्स ब्लॉक इंजेक्शन

**नियम**: `oas3-api-servers` | **कॉन्ट्रैक्ट परिवर्तन**: नहीं

उन स्पेक्स में F5 XC टेनेंट URL टेम्पलेट जोड़ता है जिनमें `servers` ब्लॉक नहीं है:

```json
"servers": [{
  "url": "https://{tenant}.console.ves.volterra.io",
  "description": "F5 Distributed Cloud API",
  "variables": {
    "tenant": {
      "default": "example-tenant",
      "description": "Your F5 XC tenant name"
    }
  }
}]
```

सर्वर URL और वेरिएबल मान [validation.yaml](/05-configuration/) में `spectral.servers` से आते हैं।

### संपर्क जानकारी इंजेक्शन

**नियम**: `info-contact` | **कॉन्ट्रैक्ट परिवर्तन**: नहीं

उन स्पेक्स के लिए `info.contact` में संपर्क जानकारी जोड़ता है जिनमें यह नहीं है:

```json
"contact": {
  "name": "F5 Distributed Cloud",
  "url": "https://docs.cloud.f5.com",
  "email": "support@f5.com"
}
```

मान [validation.yaml](/05-configuration/) में `spectral.contact` से आते हैं।

### ऑपरेशन टैग व्युत्पत्ति

**नियम**: `operation-tags` | **कॉन्ट्रैक्ट परिवर्तन**: नहीं

बिना टैग वाले ऑपरेशनों के लिए URL पाथ प्रीफ़िक्स से एक टैग व्युत्पन्न करता है। `/api/config/namespaces/{namespace}/healthchecks` जैसे पाथ के लिए, टैग `config` होता है (दूसरा गैर-पैरामीटर खंड)। टैग ऑपरेशन के `tags` ऐरे और स्पेक-स्तर `tags` सूची दोनों में जोड़ा जाता है।

### अप्रयुक्त कम्पोनेंट हटाना

**नियम**: `oas3-unused-component` | **कॉन्ट्रैक्ट परिवर्तन**: नहीं

`components.schemas` से उन कम्पोनेंट स्कीमाज़ को हटाता है जिनका स्पेक में कहीं भी संदर्भ नहीं है। किसी भी ऑपरेशन को प्रभावित किए बिना स्पेक का आकार कम करता है।

### OperationId डीडुप्लीकेशन

**नियम**: `operation-operationId-unique` | **कॉन्ट्रैक्ट परिवर्तन**: नहीं

जब कई ऑपरेशन एक ही `operationId` साझा करते हैं तो HTTP मेथड को प्रत्यय के रूप में जोड़ता है (उदाहरण: `ListItems_get`)। यह कोड जनरेटर्स के लिए प्रत्येक operationId को अद्वितीय बनाता है।

### अमान्य उदाहरण/डिफ़ॉल्ट हटाना

**नियम**: `oas3-valid-schema-example` | **कॉन्ट्रैक्ट परिवर्तन**: नहीं

जब मान स्कीमा के अपने कन्स्ट्रेंट्स (गलत प्रकार, सीमा से बाहर, आदि) के अनुरूप नहीं होता तो स्कीमाज़ से `example` या `default` मान हटाता है। कोड जनरेटर्स को अमान्य सैंपल पेलोड उत्पन्न करने से रोकता है।

### Script टैग स्ट्रिपिंग

**नियम**: `no-script-tags-in-markdown` | **कॉन्ट्रैक्ट परिवर्तन**: नहीं

regex प्रतिस्थापन का उपयोग करके `description` फ़ील्ड से `<script>` टैग हटाता है। कुछ अपस्ट्रीम स्पेक्स में इंजेक्ट किए गए script टैग होते हैं जो डॉक्यूमेंटेशन रेंडरर्स को तोड़ते हैं।

## सुरक्षा मेटाडेटा

**कॉन्ट्रैक्ट परिवर्तन**: नहीं (केवल प्रलेखन)

प्रत्येक स्पेक को एक `apiKeyAuth` सुरक्षा स्कीम प्राप्त होती है यदि पहले से मौजूद नहीं है। यह हमेशा इंजेक्ट किया जाता है चाहे Spectral ने इसे फ़्लैग किया हो या नहीं।

```json
"components": {
  "securitySchemes": {
    "apiKeyAuth": {
      "type": "apiKey",
      "in": "header",
      "name": "Authorization",
      "description": "F5 XC API Token (format: APIToken <token>)"
    }
  }
},
"security": [{ "apiKeyAuth": [] }]
```

स्कीम परिभाषा [validation.yaml](/05-configuration/) में `spectral.security_scheme` से आती है। यह API उपभोक्ताओं को प्रमाणीकरण का तरीका बताता है लेकिन रनटाइम व्यवहार नहीं बदलता।

## JSON फ़ॉर्मेटिंग

**कॉन्ट्रैक्ट परिवर्तन**: नहीं

सभी आउटपुट JSON स्पेक्स Biome-संगत ऐरे कॉम्पैक्टर (`scripts/utils/spec_loader.py` में `_compact_short_arrays`) से गुज़रते हैं। 120 वर्णों के भीतर आने वाले छोटे ऐरे एक पंक्ति में संक्षिप्त कर दिए जाते हैं:

```json
// Before
"enum": [
  "ACTIVE",
  "INACTIVE"
]

// After
"enum": ["ACTIVE", "INACTIVE"]
```

लाइन-लंबाई सीमा से अधिक वाले ऐरे बहु-पंक्ति बने रहते हैं।

## सुधार प्राथमिकता

जब कई सुधार रणनीतियाँ लागू हो सकती हैं, तो रीकन्सिलर `reconciliation.priority` में कॉन्फ़िगर की गई प्राथमिकता क्रम का उपयोग करता है:

1. **existing** -- मौजूदा स्पेक कन्स्ट्रेंट का उपयोग करें यदि वह मान्य है
2. **discovery** -- लाइव API के खोजे गए व्यवहार का उपयोग करें
3. **inferred** -- समान एंडपॉइंट्स के पैटर्न से अनुमान लगाएँ

सभी सुधार लागू होने के बाद प्रत्येक सुधारे गए स्पेक को `openapi-spec-validator` से सत्यापित किया जाता है। यदि सत्यापन विफल होता है, तो मूल स्पेक को फ़ॉलबैक के रूप में उपयोग किया जाता है और सुधार को त्रुटि के रूप में लॉग किया जाता है।

يُطبِّق المُصالِح (`scripts/reconcile.py`) أربع فئات من الإصلاحات على المواصفات الأصلية. تُغيِّر إصلاحات القيود عقود واجهة برمجة التطبيقات لتتوافق مع السلوك المُلاحَظ. أما جميع الإصلاحات الأخرى فهي متعلقة بالبيانات الوصفية فقط ولا تُعدِّل عقود واجهة برمجة التطبيقات.

## إصلاحات القيود

تُعدِّل هذه الإصلاحات قيود مخطط OpenAPI بحيث تتطابق المواصفة مع ما تُطبِّقه واجهة برمجة التطبيقات الحية فعلياً. يُحرَّك كل إصلاح بواسطة كائن `Discrepancy` يُنتَج أثناء [التحقق](/02-pipeline/#2-validate).

| الاستراتيجية | المُحفِّز | ما تفعله | تغيير العقد |
|----------|---------|--------------|-----------------|
| **تخفيف** | `SPEC_STRICTER` | المواصفة ترفض قيماً تقبلها واجهة برمجة التطبيقات. تُخفِّض `minLength`/`minimum`، وترفع `maxLength`/`maximum`، وتُضيف قيم تعداد مفقودة. | نعم -- توسِّع القيم المقبولة |
| **تشديد** | `SPEC_LOOSER` | المواصفة تقبل قيماً ترفضها واجهة برمجة التطبيقات. ترفع `minLength`/`minimum`، وتُخفِّض `maxLength`/`maximum`، وتُقيِّد التعدادات بالقيم المُلاحَظة. | نعم -- تُضيِّق القيم المقبولة |
| **إضافة** | `MISSING_CONSTRAINT` | واجهة برمجة التطبيقات تُطبِّق قيداً غير موثَّق في المواصفة. تُضيف القيد إلى المخطط. | نعم -- تُضيف قيداً جديداً |
| **إزالة** | `EXTRA_CONSTRAINT` | المواصفة تُعلن عن قيد تتجاهله واجهة برمجة التطبيقات. تُزيل القيد من المخطط. | نعم -- تُزيل القيد |

:::caution
تُغيِّر إصلاحات القيود عقد واجهة برمجة التطبيقات. تُطبَّق فقط عندما تكون مدعومة بأدلة من Schemathesis أو اختبار التحقق من القيود مقابل واجهة برمجة التطبيقات الحية.
:::

### أنواع القيود المدعومة

فئات قيود OpenAPI العشر التي يتم اختبارها وإصلاحها:

- **طول السلسلة النصية** -- `minLength`، `maxLength`
- **النمط** -- `pattern` (تعبير نمطي)
- **الحدود الرقمية** -- `minimum`، `maximum`، `exclusiveMinimum`، `exclusiveMaximum`
- **الحقول المطلوبة** -- `required`
- **قيم التعداد** -- `enum`
- **حدود المصفوفات** -- `minItems`، `maxItems`، `uniqueItems`
- **بنية الكائن** -- `additionalProperties`، `properties`، `propertyNames`
- **التركيب** -- `oneOf`، `anyOf`، `allOf`
- **التبعيات** -- `dependentRequired`، `dependentSchemas`
- **أنواع البيانات** -- `type`، `format`

## إثراءات Spectral

تُعالج هذه الإصلاحات مشكلات جودة OAS3 التي اكتشفها [فحص Spectral](/04-spectral-rules/). تُضيف أو تُصحِّح البيانات الوصفية دون تغيير سلوك واجهة برمجة التطبيقات.

### حقن كتلة الخوادم

**القاعدة**: `oas3-api-servers` | **تغيير العقد**: لا

تُضيف قالب عنوان URL لمستأجر F5 XC إلى المواصفات التي تفتقر إلى كتلة `servers`:

```json
"servers": [{
  "url": "https://{tenant}.console.ves.volterra.io",
  "description": "F5 Distributed Cloud API",
  "variables": {
    "tenant": {
      "default": "example-tenant",
      "description": "Your F5 XC tenant name"
    }
  }
}]
```

يأتي عنوان URL للخادم وقيم المتغيرات من `spectral.servers` في [validation.yaml](/05-configuration/).

### حقن معلومات الاتصال

**القاعدة**: `info-contact` | **تغيير العقد**: لا

تُضيف معلومات الاتصال إلى `info.contact` للمواصفات التي تفتقر إليها:

```json
"contact": {
  "name": "F5 Distributed Cloud",
  "url": "https://docs.cloud.f5.com",
  "email": "support@f5.com"
}
```

تأتي القيم من `spectral.contact` في [validation.yaml](/05-configuration/).

### اشتقاق وسوم العمليات

**القاعدة**: `operation-tags` | **تغيير العقد**: لا

يُشتق وسم من بادئة مسار عنوان URL للعمليات غير الموسومة. بالنسبة لمسار مثل `/api/config/namespaces/{namespace}/healthchecks`، يكون الوسم هو `config` (الجزء الثاني غير المُعامَل). يُضاف الوسم إلى كل من مصفوفة `tags` الخاصة بالعملية وقائمة `tags` على مستوى المواصفة.

### إزالة المكونات غير المستخدمة

**القاعدة**: `oas3-unused-component` | **تغيير العقد**: لا

تُزيل مخططات المكونات من `components.schemas` التي لا يُشار إليها في أي مكان في المواصفة. تُقلِّل حجم المواصفة دون التأثير على أي عملية.

### إزالة تكرار معرِّفات العمليات

**القاعدة**: `operation-operationId-unique` | **تغيير العقد**: لا

تُلحق طريقة HTTP كلاحقة (مثل `ListItems_get`) عندما تتشارك عمليات متعددة نفس `operationId`. هذا يجعل كل معرِّف عملية فريداً لمولِّدات الكود.

### إزالة الأمثلة/القيم الافتراضية غير الصالحة

**القاعدة**: `oas3-valid-schema-example` | **تغيير العقد**: لا

تُزيل قيم `example` أو `default` من المخططات عندما لا تتوافق القيمة مع قيود المخطط ذاته (نوع خاطئ، خارج النطاق، إلخ). تمنع مولِّدات الكود من إنتاج حمولات عينة غير صالحة.

### تجريد وسوم Script

**القاعدة**: `no-script-tags-in-markdown` | **تغيير العقد**: لا

تُجرِّد وسوم `<script>` من حقول `description` باستخدام استبدال بتعبير نمطي. بعض المواصفات الأصلية تحتوي على وسوم script محقونة تُعطِّل عارضات التوثيق.

## البيانات الوصفية للأمان

**تغيير العقد**: لا (توثيق فقط)

تتلقى كل مواصفة نظام أمان `apiKeyAuth` إذا لم يكن موجوداً بالفعل. يُحقَن هذا دائماً بغض النظر عمّا إذا كان Spectral قد أشار إليه.

```json
"components": {
  "securitySchemes": {
    "apiKeyAuth": {
      "type": "apiKey",
      "in": "header",
      "name": "Authorization",
      "description": "F5 XC API Token (format: APIToken <token>)"
    }
  }
},
"security": [{ "apiKeyAuth": [] }]
```

يأتي تعريف النظام من `spectral.security_scheme` في [validation.yaml](/05-configuration/). هذا يُخبر مستهلكي واجهة برمجة التطبيقات بكيفية المصادقة لكنه لا يُغيِّر السلوك أثناء التشغيل.

## تنسيق JSON

**تغيير العقد**: لا

تمر جميع مواصفات JSON الناتجة عبر ضاغط مصفوفات متوافق مع Biome (`_compact_short_arrays` في `scripts/utils/spec_loader.py`). يتم طي المصفوفات القصيرة التي تتسع في 120 حرفاً إلى سطر واحد:

```json
// Before
"enum": [
  "ACTIVE",
  "INACTIVE"
]

// After
"enum": ["ACTIVE", "INACTIVE"]
```

تبقى المصفوفات التي تتجاوز حد طول السطر متعددة الأسطر.

## أولوية الإصلاح

عندما يمكن تطبيق استراتيجيات إصلاح متعددة، يستخدم المُصالِح ترتيب أولوية مُهيَّأ في `reconciliation.priority`:

1. **existing** -- استخدام قيد المواصفة الحالي إذا كان صالحاً
2. **discovery** -- استخدام السلوك المُكتشَف لواجهة برمجة التطبيقات الحية
3. **inferred** -- الاستنتاج من الأنماط عبر نقاط النهاية المشابهة

يتم التحقق من كل مواصفة مُصلَحة باستخدام `openapi-spec-validator` بعد تطبيق جميع الإصلاحات. إذا فشل التحقق، تُستخدم المواصفة الأصلية كبديل احتياطي ويُسجَّل الإصلاح كخطأ.

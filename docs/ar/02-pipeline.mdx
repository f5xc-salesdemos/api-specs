يعمل خط الأنابيب يومياً في الساعة 6 صباحاً بتوقيت UTC عبر GitHub Actions (`validate-and-release.yml`) وعند كل دفع إلى `main` يمس `scripts/` أو `config/` أو `pyproject.toml`. كما يمكن تشغيله يدوياً مع خيارات لفرض إصدار أو تخطي اختبارات API المباشرة.

## مراحل خط الأنابيب

```
 Download         Validate          Spectral Lint
 ┌──────┐        ┌──────────┐      ┌─────────────┐
 │ Fetch │───────▶│Schemathesis│────▶│  Discover   │
 │ specs │        │Constraints│     │  violations │
 └──────┘        └──────────┘      └─────────────┘
                                          │
                 Spectral Gate       Reconcile
                ┌─────────────┐    ┌──────────┐
                │  Enforce    │◀───│Apply fixes│
                │  thresholds │    │to specs   │
                └─────────────┘    └──────────┘
                       │
                    Release
                 ┌──────────┐
                 │  Package  │
                 │  & tag    │
                 └──────────┘
```

### 1. التنزيل

يقوم `scripts/download.py` بجلب ملف ZIP الرسمي لمواصفات F5 XC OpenAPI من `docs.cloud.f5.com`. يستخدم ترويسات HTTP ETag لطلبات GET المشروطة -- إذا لم تتغير الحزمة المصدرية، يتم تخطي التنزيل. تُستخرج ملفات JSON الخاصة بالمواصفات إلى `specs/original/`.

### 2. التحقق

ينظم `scripts/validate.py` نوعين من الاختبارات مقابل واجهة F5 XC API المباشرة:

- **Schemathesis** -- اختبار قائم على الخصائص يولّد حمولات عشوائية صالحة/غير صالحة وفقاً للمواصفات ويتحقق مما إذا كانت الواجهة البرمجية تتوافق مع قيود المواصفات.
- **التحقق من القيود** -- اختبارات مستهدفة لجميع فئات قيود OpenAPI العشر (طول النص، النمط، الحدود الرقمية، الحقول المطلوبة، القيم المحددة، حدود المصفوفات، بنية الكائنات، التركيب، التبعيات، أنواع البيانات).

تُكتب التناقضات في `reports/validation_report.json`.

### 3. فحص Spectral (الاكتشاف)

يقوم `scripts/spectral_lint.py --mode discover` بتشغيل واجهة Spectral CLI مقابل المواصفات الأصلية باستخدام `spectral-pipeline.yaml`. يكتشف هذا مشاكل جودة OAS3 -- كتل الخوادم المفقودة، معلومات الاتصال المفقودة، العمليات غير المصنفة، مخططات المكونات غير المستخدمة، معرفات العمليات المكررة، الأمثلة غير الصالحة، ووسوم السكريبت في الأوصاف.

تُحوّل المخالفات إلى كائنات `Discrepancy` وتُكتب في `reports/spectral_report.json`. راجع [قواعد Spectral](/04-spectral-rules/) للاطلاع على مجموعة القواعد الكاملة.

### 4. المواءمة

يستهلك `scripts/reconcile.py` ملفي التقارير ويطبق الإصلاحات على كل مواصفة:

- **إصلاحات القيود** تعدّل قيود المخططات (تخفيف، تشديد، إضافة، إزالة) بناءً على سلوك API المُلاحظ. راجع [الإصلاحات المطبقة](/03-fixes-applied/).
- **إصلاحات Spectral** تُثري المواصفات بالخوادم ومعلومات الاتصال والوسوم وبيانات الأمان الوصفية، وتزيل المكونات الميتة.
- **بيانات الأمان الوصفية** تُحقن دائماً إذا كان تكوين `security_scheme` موجوداً، حتى عندما لا يشير Spectral إلى وجود مشكلة.

تُكتب المواصفات المُصلحة في `release/specs/` وتُتحقق باستخدام `openapi-spec-validator` قبل الحفظ. إذا فشلت مواصفة مُصلحة في التحقق، تُستخدم النسخة الأصلية كبديل.

### 5. بوابة Spectral

يعيد `scripts/spectral_lint.py --mode gate` تشغيل Spectral مقابل المواصفات المُوائمة في `release/specs/`. تتحقق البوابة من عدد الأخطاء والتحذيرات مقابل عتبات قابلة للتكوين (`spectral.gate.max_errors` و `spectral.gate.max_warnings` في `validation.yaml`).

:::note
المرحلة 1 (الحالية): العتبات هي `null` (تحذير فقط). اضبط `max_errors: 0` لفرض بوابة خالية من الأخطاء.
:::

### 6. الإصدار

يقوم `scripts/release.py` بتعبئة المواصفات المُوائمة في أرشيف ZIP مُرقّم بالإصدار. تستخدم الإصدارات التنسيق `YYYY.MM.DD-<patch>` المُستمد من تاريخ بيانات المواصفات المصدرية الوصفية. يتضمن الإصدار:

- ملفات مواصفات النطاقات الفردية (268 ملف JSON)
- ملف `openapi.json` مدمج يجمع جميع النطاقات
- ملف `CHANGELOG.md` يسرد كل إصلاح تم تطبيقه
- ملخص `VALIDATION_REPORT.md`

تنشئ مهمة CI إصداراً على GitHub مع إرفاق ملف ZIP، مُوسوماً بـ `v<version>`. يمنع تجزئة المحتوى الإصدارات المكررة عندما لا يتغير شيء.

## تحديث التوثيق

تقوم مهمة CI موازية بإنشاء `docs/01-validation-report.mdx` من تقارير التحقق باستخدام `scripts/generate_docs.py`، ثم تفتح طلب سحب إذا تغير المحتوى.

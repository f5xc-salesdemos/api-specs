يعمل خط الأنابيب يوميًا في الساعة 6 صباحًا بتوقيت UTC عبر GitHub Actions (`validate-and-release.yml`) وعند كل عملية push إلى فرع `main` تمس `scripts/` أو `config/` أو `pyproject.toml`. يمكن أيضًا تشغيله يدويًا مع خيارات لفرض إصدار أو تخطي اختبارات API المباشرة.

## مراحل خط الأنابيب

```
 Download         Validate          Spectral Lint
 ┌──────┐        ┌──────────┐      ┌─────────────┐
 │ Fetch │───────▶│Schemathesis│────▶│  Discover   │
 │ specs │        │Constraints│     │  violations │
 └──────┘        └──────────┘      └─────────────┘
                                          │
                 Spectral Gate       Reconcile
                ┌─────────────┐    ┌──────────┐
                │  Enforce    │◀───│Apply fixes│
                │  thresholds │    │to specs   │
                └─────────────┘    └──────────┘
                       │
                    Release
                 ┌──────────┐
                 │  Package  │
                 │  & tag    │
                 └──────────┘
```

### 1. التنزيل

يقوم `scripts/download.py` بجلب ملف ZIP الرسمي لمواصفات F5 XC OpenAPI من `docs.cloud.f5.com`. يستخدم ترويسات HTTP ETag لطلبات GET المشروطة -- إذا لم تتغير الحزمة المصدرية، يتم تخطي التنزيل. يتم وضع ملفات JSON المستخرجة في `specs/original/`.

### 2. التحقق

يقوم `scripts/validate.py` بتنظيم نوعين من الاختبارات مقابل واجهة F5 XC API المباشرة:

- **Schemathesis** -- اختبار قائم على الخصائص يولّد حمولات عشوائية صالحة/غير صالحة وفقًا للمواصفات ويتحقق مما إذا كانت واجهة API متوافقة مع قيود المواصفات.
- **التحقق من القيود** -- اختبارات موجهة لجميع فئات قيود OpenAPI العشر (طول النص، النمط، الحدود الرقمية، الحقول المطلوبة، التعدادات، حدود المصفوفات، بنية الكائنات، التركيب، التبعيات، أنواع البيانات).

يتم كتابة التناقضات في `reports/validation_report.json`.

### 3. فحص Spectral (الاكتشاف)

يقوم `scripts/spectral_lint.py --mode discover` بتشغيل واجهة Spectral CLI على المواصفات الأصلية باستخدام `spectral-pipeline.yaml`. يكتشف هذا مشكلات جودة OAS3 -- كتل الخوادم المفقودة، معلومات الاتصال المفقودة، العمليات غير المُوسومة، مخططات المكونات غير المستخدمة، معرّفات العمليات المكررة، الأمثلة غير الصالحة، وعلامات النصوص البرمجية في الأوصاف.

يتم تحويل المخالفات إلى كائنات `Discrepancy` وكتابتها في `reports/spectral_report.json`. راجع [قواعد Spectral](/04-spectral-rules/) للاطلاع على مجموعة القواعد الكاملة.

### 4. المطابقة

يستهلك `scripts/reconcile.py` ملفي التقارير ويطبق الإصلاحات على كل مواصفة:

- **إصلاحات القيود** تعدّل قيود المخططات (تخفيف، تشديد، إضافة، إزالة) بناءً على سلوك API الملاحظ. راجع [الإصلاحات المطبقة](/03-fixes-applied/).
- **إصلاحات Spectral** تُثري المواصفات بالخوادم ومعلومات الاتصال والعلامات وبيانات الأمان الوصفية، وتزيل المكونات غير المستخدمة.
- **بيانات الأمان الوصفية** تُحقن دائمًا إذا كان إعداد `security_scheme` موجودًا، حتى لو لم يُشر Spectral إلى ذلك.

تُكتب المواصفات المُصلحة في `release/specs/` ويتم التحقق منها باستخدام `openapi-spec-validator` قبل الحفظ. إذا فشل التحقق من مواصفة مُصلحة، يتم استخدام المواصفة الأصلية كبديل احتياطي.

### 5. بوابة Spectral

يقوم `scripts/spectral_lint.py --mode gate` بإعادة تشغيل Spectral على المواصفات المُطابقة في `release/specs/`. تتحقق البوابة من أعداد الأخطاء والتحذيرات مقابل عتبات قابلة للتكوين (`spectral.gate.max_errors` و `spectral.gate.max_warnings` في `validation.yaml`).

:::note
المرحلة 1 (الحالية): العتبات هي `null` (تحذير فقط). عيّن `max_errors: 0` لفرض بوابة بدون أخطاء.
:::

### 6. الإصدار

يقوم `scripts/release.py` بتعبئة المواصفات المُطابقة في أرشيف ZIP مُرقّم الإصدار. تستخدم الإصدارات التنسيق `YYYY.MM.DD-<patch>` المُشتق من تاريخ البيانات الوصفية للمواصفات المصدرية. يتضمن الإصدار:

- ملفات مواصفات النطاقات الفردية (268 ملف JSON)
- ملف `openapi.json` مدمج يجمع جميع النطاقات
- ملف `CHANGELOG.md` يسرد كل إصلاح مُطبّق
- ملخص `VALIDATION_REPORT.md`

تقوم مهمة CI بإنشاء إصدار GitHub مع إرفاق ملف ZIP، مُوسومًا بـ `v<version>`. يمنع تجزئة المحتوى الإصدارات المكررة عندما لا يتغير شيء.

## تحديث التوثيق

تقوم مهمة CI موازية بإنشاء `docs/01-validation-report.mdx` من تقارير التحقق باستخدام `scripts/generate_docs.py`، ثم تفتح طلب PR إذا تغير المحتوى.
